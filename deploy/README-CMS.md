# Deploying the Alayay site + CMS

## Architecture

Two PM2 apps on the same VPS, both behind nginx:

| App | PM2 name | Port | Domain | What it is |
|---|---|---|---|---|
| Website | `alayay` | 3013 | `alayay.com` | Next.js site |
| CMS | `alayay-cms` | 3014 | `cms.alayay.com` | Express — serves the admin UI (`cms/dist`) **and** the `/api` endpoints |

The site reads content from the CMS over loopback (`CMS_URL=http://127.0.0.1:3014`),
at build/revalidate time. Port 3014 is never exposed directly — nginx fronts it.

**How content reaches the site:** pages use `getStaticProps` + `revalidate: 60`
(ISR), so a CMS edit appears on the site **within ~60 seconds**. No rebuild, no
redeploy. If the CMS is down or a field is blank, the site falls back to its
built-in content — it never breaks or renders empty.

## Before the first deploy

1. **DNS** — add an A record for `cms.alayay.com` pointing at the server
   (same IP as the main site). If using Cloudflare, set it to Full mode; the
   self-signed cert generated by `setup-server.sh` is fine behind that.

2. **Secrets** — on the server, export these before the first `pm2 start`, and
   persist them (e.g. in `/root/.bashrc` or a systemd env file):

   ```bash
   # Session signing key — any long random string. Changing it logs everyone out.
   export CMS_JWT_SECRET="$(openssl rand -hex 32)"

   # OPTIONAL: seed the initial admin password (bcrypt hash).
   # Generate on any machine with the cms deps installed:
   #   node -e "console.log(require('bcryptjs').hashSync('YOUR_PASSWORD', 10))"
   export CMS_ADMIN_PASSWORD_HASH='$2a$10$...'
   ```

   If `CMS_ADMIN_PASSWORD_HASH` is unset the default password is `alayay2026`
   — **change it immediately** via the CMS → Account page. Once changed it is
   written to `cms/content/_auth.json`, which takes precedence over the env var.

## Deploy

```bash
# First time — installs nginx vhosts, certs, deps, builds, starts both PM2 apps
bash deploy/deploy.sh --init

# Routine redeploy — builds site + CMS locally, rsyncs, restarts both apps
bash deploy/deploy.sh
```

## Data that lives ONLY on the server

These are the client's work. `deploy.sh` explicitly excludes them from rsync so
`--delete` can never wipe them:

- `cms/content/` — all edited content (`settings.json`, `hero.json`, …) plus
  `_auth.json` (the admin password hash)
- `cms/public/uploads/` — every image uploaded through the CMS

They are also in `.gitignore`. **Do not remove those rsync excludes** — a deploy
would erase everything the client has entered.

## Images

Images uploaded in the CMS are saved under `cms/public/uploads/` and referenced
in content as `/uploads/<file>`. The site's nginx vhost proxies `/uploads/` to
the CMS process, so those URLs resolve on the main domain. If you ever move the
CMS off this box, that `location /uploads/` block must follow it.

## What the CMS can and cannot edit

**Editable:** hero (badge/headings/description/CTA labels + background image),
service titles/descriptions/images, project titles/categories/descriptions +
before/after images, testimonials (name/role/text/rating/avatar), and site
settings (company name, phone, WhatsApp, email, address, socials, SEO).

**Still hardcoded** in `src/locales/translations.ts` / `src/components/alayay/data.ts`:
hero feature badges and the third heading line, About page, AMC plans,
How It Works, Why Us, and footer links/newsletter. These fall back safely; to
make them editable the CMS schema needs extending.
